How I use AI as a product owner

Most of the AI conversation in tech is about writing code. This page is about the rest of the work: how AI can help reshape planning, triage, synthesis, and execution for people whose day-to-day job is not software development.

Most AI discussion in tech starts with code generation. That makes sense. It is visible, easy to demo, and easy for engineers to judge. It is also the use case most people recognize first.

My point here is different. I do not write code day to day. I spend more time turning messy inputs into usable decisions, plans, artifacts, and follow-through. So when I think about AI at work, I am less interested in whether it can replace a developer for five minutes and more interested in how it can improve the workflows around the rest of the business.

Core idea: the model does the reasoning, but the real value shows up when the workflow, tools, and systems around it are designed to make that reasoning usable.

The day-to-day workflow

The workflow works because each stage has a different job. First I capture the thinking before I over-edit it. Then I use a strong model to interpret it. After that, the agent layer, skills, MCP connections, and other tools turn it into repeatable output I can actually use.

01

Capture the problem while it is still fresh

I usually start by talking through the problem instead of trying to type a perfect request. Voice works well for me, but the real point is preserving the raw thinking before it gets flattened. That input might be a transcript, meeting notes, screenshots, or half-formed requirements.

02

Use the model to shape the mess

Once the rough thought exists, I pass it into the agent setup that fits the job. The model is still the reasoning engine, but now it is working on something closer to the real problem instead of a tiny polished prompt that already lost context.

03

Connect it to the systems that matter

This is where skills, MCP connections, and other tools matter. They stop the model from improvising every time and give it guardrails for the artifact I want, the systems it can inspect, and the output shape it should follow.

Coding is the obvious use case, but not the only one

In software engineering circles, AI usually enters the conversation through code generation. People can see it, compare it, and argue about which model writes better code. That is why it gets so much attention.

But most businesses do not run only on code. They run on planning, communication, decisions, triage, documentation, coordination, and handoffs between people and systems. If AI only gets framed as a coding assistant, people miss the much broader opportunity to improve how work moves through an organization.

Where Claude or Codex comes in

Claude or Codex is the reasoning layer in this workflow. It can interpret rough notes, spot patterns, tighten language, and help me move from a messy thought to a cleaner draft. On its own, though, a model is still mostly a brain without much reach.

The output gets much better when the model is paired with the right delivery system. I care less about model tribalism and more about whether the full setup can turn messy input into a clear artifact without flattening the intent behind it.

Why the skills matter

The skills are what keep this from turning into generic AI writing. I do not think of them as one-off prompts. I think of them as reusable guardrails for recurring work. A good skill tells the agent what kind of artifact it is creating, what context matters, what structure to follow, and what kind of drift to avoid.

Some of those skills are tied to Jira artifacts because that is a big part of my day-to-day workflow. But the point is broader than Jira. When I find a task I repeat often enough, I want more than a smart model. I want a repeatable setup that helps the model produce the right kind of output more consistently.

Why the agent layer matters

The model alone can draft text. The agent layer is what lets me offload repetitive work. It is the difference between asking for a nice-sounding paragraph and asking a system to turn rough notes into the right Jira artifact with the right structure, guardrails, and access to the right context.

That is the bigger pattern I care about. When I find repeatable work worth offloading, I do not just save a prompt. I build a better handoff so the model can do useful work inside a real process.

What this means for businesses and software providers

This shift is not only about how individuals work. It also changes what makes business software more valuable. If a product cannot be accessed programmatically, it gets harder to include in modern AI workflows.

Over time, tools with usable APIs, structured exports, and predictable ways to read or write data will become more appealing than tools that trap users inside manual interfaces. AI works best when it can participate in a workflow, not when it has to stop at the edge of the screen.

I think more product teams should be asking a simple question: if a customer wants to use our product through an agent, automation, or AI assistant, have we made that possible? If not, a competitor eventually will.

The exact skill changes based on the artifact I need, but the handoff pattern stays pretty similar. Capture the messy input, name the kind of output I want, load the right guardrails, then review and refine.

What this changes in practice

The biggest gain is not just speed. It is being able to offload the repetitive translation work that sits between an idea and a usable artifact. I still own the thinking, prioritization, and final call. I just spend less time rebuilding the same structure over and over again.

It also changes how I evaluate tools. I care more about whether a product can participate in an AI-assisted workflow than whether it simply has a chatbot bolted onto the side. If a tool exposes the right data and actions, it becomes much easier to include in the way people will actually work.

What is still evolving

This workflow is still getting better. I want to add tighter examples, cleaner side-by-side comparisons between rough input and final output, and direct links to example skills and prompt patterns without turning the page into a giant list.

For now, this page is a first draft of the larger point: AI is not only about asking smarter questions or writing more code. It is about redesigning workflows so useful work moves faster with better structure.

Bottom line

The most useful AI workflow I have found as a product owner is not "open a chatbot and ask a smart question." It is identifying repeatable work worth offloading, then building a reliable path from messy thinking to the exact artifact I need next.

Capture helps. The model helps. But the lasting value comes from adapting the workflow, connecting the tools, and making the surrounding systems usable by agents and AI.
